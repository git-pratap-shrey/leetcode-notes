# My Solution
~~~cpp
class Solution {
public:
    vector<int> findGoodIntegers(int n) {

        int size = cbrt(n) + 1;
        
        vector<int> cube(size);

        for(int i = 0; i < size; i++){
            cube[i] = (i+1) * (i+1) * (i+1);
            // cout<<cube[i]<<endl;
        }

        unordered_set<int> hash;
        unordered_set<int> answerSet;
        
        for(int i = 0; i < size; i++){
            for(int j = i; j < size; j++){
                if(cube[i] + cube[j] > n){
                    break;
                }
                if(hash.count(cube[i] + cube[j])){
                    answerSet.insert(cube[i] + cube[j]);
                }
                else{
                    hash.insert(cube[i] + cube[j]);
                }
            }
        }
        vector<int> answer(answerSet.begin(), answerSet.end());
        sort(answer.begin(), answer.end());
        return answer;
    }
};
~~~

# Submission Review
## Approach
*   **Technique:** Brute force using precomputed cubes and a hash-based lookup to find values expressible as a sum of two cubes in at least two distinct ways.
*   **Optimality:** Optimal for the given approach. The logic correctly handles the "multiple ways" requirement by tracking sums in `hash` and promoting duplicates to `answerSet`.

## Complexity
*   **Time Complexity:** $O(k^2)$, where $k$ is the number of cubes less than $n$ ($k \approx \sqrt[3]{n}$). Given $k \approx n^{1/3}$, the complexity is $O(n^{2/3})$.
*   **Space Complexity:** $O(n^{2/3})$ to store the sums in `unordered_set`.

## Efficiency Feedback
*   **Bottleneck:** The use of `unordered_set` has a constant time overhead for hashing. Given that the number of pairs $(i, j)$ is relatively small for reasonable $n$, a simple `std::vector` or a flat array (if the range is known) would be faster due to cache locality.
*   **Optimization:** The inner loop condition `if(cube[i] + cube[j] > n) break;` is efficient. However, the `size` calculation using `cbrt(n) + 1` is correct but slightly tight; ensure it accounts for edge cases where $n$ is very small.

## Code Quality
*   **Readability:** Good. The code is concise and the logic is easy to follow.
*   **Structure:** Good. The separation of precomputing cubes and the double-loop search is clear.
*   **Naming:** Moderate. `hash` is a generic name; `sums` or `seenSums` would be more descriptive. `size` is slightly ambiguous (could refer to the number of cubes or the value limit).
*   **Improvements:**
    *   **Data Structures:** If memory is a constraint or performance is critical, replace `unordered_set` with a `std::vector<int>` and sort it to find duplicates, or use a `std::vector<bool>` if the range of sums is small and known.
    *   **Redundancy:** The `answerSet` is unnecessary if you only insert into `answer` when a collision is detected (and handle subsequent hits carefully), though the current approach is safe.
    *   **Safety:** The `cube` vector initialization assumes `i+1` index. While functional, it is cleaner to index directly from `1` to `size` or use `i` in the loop.

# Question Revision
### Revision Report: Integers With Multiple Sum of Two Cubes

**Pattern:** Hash Map / Frequency Counting

**Brute Force:**
Iterate through all pairs $(i, j)$ such that $i^3 + j^3 = N$, checking if $N$ can be represented as a sum of two cubes in more than one way. This involves nested loops up to $\sqrt[3]{N}$, resulting in $O(N^{2/3})$ time.

**Optimal Approach:**
1. Precompute/iterate through all pairs $(i, j)$ where $i^3 + j^3 = S$.
2. Use a Hash Map to store the frequency of each sum $S$ encountered.
3. If a sum $S$ appears more than once, add it to the result set.
4. **Complexity:** $O(n^{2/3})$ time and $O(n^{2/3})$ space, where $n$ is the upper bound of the input range.

**The 'Aha' Moment:**
When the problem asks for "multiple ways" to satisfy a mathematical constraint, mapping results to frequencies rather than searching for specific matches allows you to detect collisions instantly.

**Summary:** 
Whenever you need to count how many times a specific value can be generated by a combination of inputs, use a Hash Map to store frequencies of the sums instead of searching for existing solutions.
